# Repository Guidelines

## 项目结构
本仓库的核心项目是 `poker_game/`。主要代码位于 `poker_game/Classes/`：
- `model/`：游戏状态、牌堆、布局配置、覆盖关系计算
- `controller/`：玩家操作、匹配规则、撤销与流程控制
- `view/`：卡牌显示、主牌区、顶部牌区与交互表现
- `config/`：运行时 JSON 配置
- `logging/`：启动日志与运行日志

资源文件在 `poker_game/Resources/`，其中 `config/layouts/` 存放玩法布局模板。设计文档在 `docs/`，历史实验代码在 `legacy_projects/`。

## 构建与运行
- `cmake -S poker_game -B poker_game/build`：生成或刷新构建文件
- `cmake --build poker_game/build -j4`：编译游戏并复制资源
- `./run_poker_game.sh`：从仓库根目录启动本地构建

当前没有独立自动化测试；提交前至少保证可以成功编译并手动验证相关玩法。

## 玩法开发规范
- 新玩法优先通过 `Resources/config/layouts/*.json` 配置，不要把布局写死在场景代码里。
- 布局 JSON 应明确表达结构信息，如 `mainPileCount`、`pileIndex`、`layer`，避免依赖静态 `covers`。
- `covers` 应由运行时根据层级与几何关系动态生成；修改覆盖判定时，优先改 `model/LayoutConfig.cpp`，不要分散到视图层。
- 新规则放在 `model/` 或 `controller/`，不要把规则判断塞进 `view/`。
- 扩展新牌型、新布局、新匹配逻辑时，优先增加配置和独立函数，避免直接改已有流程分支。

## 编码与维护约定
- C++ 使用 4 空格缩进，延续现有大括号换行风格。
- 类型名使用 `CamelCase`，JSON 字段使用语义清晰的英文小驼峰或现有风格，如 `mainPileCount`。
- 涉及尺寸、层级、覆盖关系的逻辑，必须单点维护，避免同一规则在多个文件重复实现。
- 能从资源真实尺寸或配置推导的数据，不要再写死常量。

## 调试与日志
- 每次启动都会生成一份日志文件，优先用日志排查启动、布局加载、覆盖计算和关卡切换问题。
- 修改布局或覆盖逻辑后，需验证：
  - 哪些牌初始翻开
  - 每张牌的 `covers` 是否符合预期
  - 不同关卡切换后主牌区是否完整刷新

## 提交与变更说明
- 提交信息使用简短祈使句，例如：`调整 fan 布局层级`、`修正动态 covers 计算`
- 涉及玩法或 UI 变化时，补充说明：
  - 修改了哪些文件
  - 改动影响的玩法规则
  - 编译与手动验证结果
## 核心原则
- 极简原则，所有代码都要以最简单的形式实现就行，完全不需要多余的一些操作，不要加太多兜底操作
- 要求代码的可维护性和可扩展性
